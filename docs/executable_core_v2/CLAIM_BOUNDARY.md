# Claim boundary

## Phase 0 claim

This phase freezes a theorem-to-runtime refinement contract. It does not implement
or validate the production runtime checker, generator, promotion controller,
PyTorch backend, or external benchmark adapter.

The defensible Phase 0 claim is:

> The repository contains a versioned, machine-readable, fail-closed contract that
> maps the completed Formal Core v2 Gate A, Gate B, selected Gate C, and RCLM
> objects to future executable schemas, immutable Python type names, runtime
> functions, certificate evidence, conformance tests, numerical semantics,
> canonical hashes, and promotion rules.

## First executable reference claim

After the later runtime phases pass their exit criteria, the strongest initial
executable claim may be:

> A finite, independently replayable theorem-to-runtime reference witness for the
> declared Gate B binary and selected Gate C commuting/diagonal scopes, with an
> untrusted proposal source, fail-closed checker, Lean-backed conformance, immutable
> package provenance, and atomic promotion or rollback.

That claim requires actual implementation and evidence. It is not established by
this contract alone.

## Included formal scope

The contract includes:

```text
Gate A conditional one-step obligations
Gate A finite accepted-trajectory composition
Gate A recovery and monitor interfaces
Gate B exact finite distributions, Shannon entropy, and support-aware KL
Gate B zero-coordinate conservative extension and exact recovery
Gate B binary improvement and stability reference
Gate C complex diagonal density-matrix construction
Gate C spectral von Neumann entropy and support-aware diagonal QRE
Gate C identity and basis-swap channels
Gate C exact selected recovery and finite trajectory
RCLM state, update, certificate, checker, recovery, monitor, and engine refinement
```

## Excluded formal scope

The contract does not expand the formal theorem surface to:

```text
arbitrary noncommuting density matrices
arbitrary CPTP maps
general matrix-logarithm quantum relative entropy
general quantum data processing
trace-distance, Petz, or approximate recovery
infinite-dimensional operator algebras
exact full Paper I semantic identity
exact full Paper II semantic identity
```

## Excluded executable claims

Neither Phase 0 nor the first reference runtime may claim:

```text
arbitrary learned-system entry
unbounded generator completeness
unbounded proof-search completeness
strict useful improvement at every recursive step
autonomous recursive self-improvement
unbounded-horizon empirical proof
frontier-scale validation
benchmark superiority
model safety in unrestricted environments
```

## Strict-improvement boundary

The current formal classical and selected quantum reference trajectories each prove
one strict information-theoretic improvement followed by accepted stability. They
do not prove a useful strict successor exists at every future step.

Therefore the runtime must distinguish:

```text
accepted continuation
strict certified improvement
successor availability
strict successor availability
```

None implies the others unless the relevant explicit evidence is present.

## Learned-system boundary

A later PyTorch candidate is an untrusted package proposal. A changed weight hash
and held-out score improvement do not by themselves establish RCLM-to-RCP
refinement. The model package, training process, evaluation evidence, protected
metrics, resource use, and rollback data must all pass the same checker and
provenance rules.

## Benchmark boundary

External benchmark results begin only after:

```text
checker conformance
adversarial rejection
reference generator
promotion controller
independent replay
```

A benchmark score cannot substitute for theorem-to-runtime refinement. Conversely,
a formal accept result does not imply external capability improvement.

## Historical artifacts

Historical v1 Python artifacts and benchmark records remain historical evidence
only. They are not automatically imported into Executable Core v2, and they do not
satisfy any v2 conformance test unless explicitly remapped and revalidated under
this contract.

## Required wording

Until the complete runtime evidence exists, repository summaries must use:

```text
contract frozen
runtime not yet implemented
runtime refinement not yet established
```

They must not use:

```text
executable RSI proved
Python checker formally verified
self-improvement demonstrated
unbounded RSI instantiated
```
