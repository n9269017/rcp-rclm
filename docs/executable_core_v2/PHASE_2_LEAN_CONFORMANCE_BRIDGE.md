# Executable Core v2 — Phase 2 pinned Lean conformance bridge

## Purpose

Phase 2 establishes an initial theorem-to-runtime conformance bridge between the
Phase 1 deterministic Python bedrock and the pinned Formal Core v2 Lean project.
It does not authorize production candidate acceptance.

The bridge follows this sequence:

```text
canonical reference packet
→ deterministic Python interpretation
→ restricted generated Lean certificate source
→ mandatory source guard
→ pinned Lake/Lean invocation
→ structured Lean verdict marker
→ Python/Lean differential comparison
→ immutable evidence report
```

## Supported reference scopes

The initial bridge covers the selected finite references already formalized in Lean:

```text
Gate B classical binary reference
  initial + improve + target + improvement certificate → accepted
  target + stay + target + stability certificate       → accepted
  wrong successor, wrong certificate, malformed packet → rejected

Gate C diagonal quantum reference
  source + swap + target + improvement certificate → accepted
  target + stay + target + stability certificate   → accepted
  wrong successor, wrong certificate, malformed packet → rejected
```

Each generated certificate proves the closed Boolean result twice: once for the RCP
reference checker and once for the canonical RCLM wrapper checker. Accepted cases
also instantiate the complete existing RCP `StepObligations` theorem. The source
checks declared RCP and RCLM theorem surfaces, so a missing or renamed declaration
fails compilation.

## Trust boundary

Python is responsible for:

```text
strict packet parsing
canonical hashing
reference interpretation
source generation from a closed grammar
source hygiene enforcement
subprocess orchestration
evidence preservation
structured verdict validation
differential comparison
```

Lean is responsible for elaborating and checking the closed reference proposition.
A successful bridge report requires both clean compilation and exact agreement
between the Python interpretation and the Lean-emitted RCP and RCLM results.

The generator, candidate code, model weights, optimizer, candidate certificate
assertions, and candidate-reported scores remain untrusted and are not part of this
phase.

## Source-generation restriction

The bridge does not accept arbitrary Lean text from a candidate. A
`LeanReferencePacket` selects only from fixed Gate B or Gate C constructors. The
source generator emits imports, a closed candidate value, a closed Boolean checker
proposition, accepted-case obligation evidence, theorem-surface checks, and one
structured verdict marker.

Before the compiler is invoked, the generated bytes are scanned for:

```text
sorry
sorryAx
admit
project-local axiom declarations
invalid UTF-8
```

A source-guard failure prevents compiler invocation. Its structured report records
the frozen guard version, source path, exact source SHA-256, byte count, matched
token and reason code, and one-based line and column. When no semantic path is
supplied, the guard uses a deterministic content-addressed virtual path.

## Pinned project identity

The compiler bridge verifies that the Formal Core paths are unchanged from the
frozen formal-source Git commit and that the relevant worktree paths are clean. It
then verifies:

```text
lean/rcp_rclm_formal_core_v2/lean-toolchain
lean/rcp_rclm_formal_core_v2/lake-manifest.json
lean/rcp_rclm_formal_core_v2/lakefile.toml
```

The required pins are:

```text
Lean:    leanprover/lean4:v4.31.0
mathlib: fabf563a7c95a166b8d7b6efca11c8b4dc9d911f
```

The bridge records stable pin hashes and a platform-specific runtime identity
containing the observed Lean and Lake version strings.

## Evidence layout

The Phase 2 runner creates:

```text
project_pin.json
phase_2_conformance_report.json
cases/<case-id>/
  packet.json
  generated_certificate.lean
  source_guard.json
  compilation.json
  toolchain_identity.json
  lean_stdout.txt
  lean_stderr.txt
  lean_verdict.json
  bridge_report.json
```

Raw stdout and stderr are preserved. The structured result is accepted only if one
and only one canonical verdict marker is present and all packet, scope, theorem
surface, layer-agreement, and acceptance fields match.

## Differential claim

A successful Phase 2 run establishes only:

> For the ten declared finite Gate B and selected Gate C reference and mutation
> cases, the deterministic Python reference interpretation equals the RCP and
> canonical RCLM results checked by the pinned Lean project.

It does not establish a general Python implementation of every Lean theorem. It does
not prove production checker soundness, arbitrary RCLM packet refinement, learned
system entry, generator completeness, promotion correctness, or recursive
self-improvement.

## Mature bridge boundary

The mature design—an independently built pinned Lean executable that parses a
restricted canonical packet directly—is not implemented in this phase. Phase 2 uses
the initial generated-source bridge so the exact generated proposition and compiler
evidence remain inspectable.
